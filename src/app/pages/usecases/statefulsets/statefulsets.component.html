Using OpenEBS for StatefulSets

Replication by application

Stateful applications on Kubernetes deployed as “statefulset” usually manage the data replication themselves. It is optional to have the replication at storage layer. In this case, OpenEBS volumes will have a single replica.

StatefulSet example applications

  • MongoDB
  • MySql with Galera replication
  • Cassandra
  • Zoo Keeper

OpenEBS vs LocalPV for StatefulSet

There are multiple advantages of deploying StatefulSet with OpenEBS PVs as OpenEBS is an integrated solution created to provide answers to storage operations. For example,

When compared to LocalPV, OpenEBS volumes are scheduled in co-ordination with Maya-ApiServer, because of which, it is easier to configure the volumes to be scheduled across availability zones.

OpenEBS volumes when configured with one replica, it is possible to change the configuration to scale the replicas from ONE to THREE if required.

Volumes can be resized easily

Upgrade of software associated with the volumes is simple and easier with OpenEBS

Deploying OpenEBS storage for a StatefulSet on public clouds

Statefulset is used primarily for application doing data replication themselves. High availability of the application is the key goal here. On public clouds operators make use of Availability Zones or AZs to achieve high availability. When application pods of a StatefulSet are scheduled across AZs, corresponding PVs also need to be scheduled across AZs. OpenEBS supports this scenario very well.

For documentation on how to configure OpenEBS replicas to be scheduled across zones, https://docs.openebs.io
